# سجل تحسينات جودة الكود — van-app

هذا الملف يوثّق بدقة ما تم تعديله في هذه النسخة، وما لم يُلمس، حتى تكون
المراجعة سهلة قبل الدمج مع فرعكم الأساسي.

## ✅ ما تم تنفيذه

1. **فصل منطق البيانات عن الواجهة** (hooks جديدة):
   - `src/hooks/useVanSale.js` — كل منطق SellTab (مخزون الكاميو، العروض، السلة، إتمام البيع بوضعَي الاتصال/عدم الاتصال). SellTab.jsx صار عرضًا فقط.
   - `src/hooks/useLoadStock.js` — كل منطق LoadTab (بحث بترقيم، تحميل صنف من المخزون الرئيسي).
   - `src/hooks/useSettlement.js` + `src/lib/settlementCalc.js` — فصل جلب بيانات التسوية عن دوال الحساب الصرفة (قابلة للاختبار بمعزل).
   - **لم يتغيّر أي سلوك وظيفي** — نفس الاستعلامات، نفس أسماء RPC، نفس رسائل التوست بالضبط.

2. **اختبارات Vitest** (`src/lib/__tests__/`):
   - `promotions.test.js` — 7 اختبارات لكل أنواع العروض (percent, fixed, bogo, tier_discount, انتهاء الصلاحية، عدم تجاوز المجموع).
   - `settlementCalc.test.js` — 4 اختبارات لحسابات ملخص التسوية.
   - `totp.test.js` — 4 اختبارات للتحقق من صحة/خطأ كود TOTP.
   - تشغيلها: `npm run test`

3. **تنظيف console.log من نسخة الإنتاج**:
   - `vite.config.js` يحذف كل `console.*`/`debugger` تلقائيًا في `npm run build` فقط (يبقى يعمل عادي في `npm run dev`).

4. **jsconfig.json + JSDoc حقيقي**: تفعيل فحص أنواع تدريجي على مستوى المحرر، مع `@typedef` فعلية لأشكال البيانات الأساسية (`VanStockItem`, `CartItem`, `SaleRow`) وتوثيق `@param`/`@returns` على الدوال الرئيسية في `useVanSale.js` و`useSettlement.js` — هذا يجعل `checkJs` يكتشف أخطاء فعلية (مثل استخدام حقل غير موجود)، وليس مجرد ملف إعداد خامل.

5. **React Query لكل استعلامات/عمليات الكتابة**:
   - `useVanSale.js`: مخزون الكاميو والعروض صارا `useQuery` بدل `useState`+`useEffect` يدوي؛ التخزين المؤقت مشترك بين `SellTab` و`LoadTab` بنفس المفتاح (`['van-stock', employeeId]`) فلا يُجلب المخزون مرتين لو فُتح التبويبان بنفس الجلسة.
   - `useLoadStock.js`: البحث بالتمرير اللانهائي صار `useInfiniteQuery` (النمط الجاهز في React Query لأزرار "تحميل المزيد") بدل إدارة `hasMore`/`loadingMore` يدوياً.
   - `useSettlement.js`: صار `useQuery` بسيط.
   - إتمام البيع (`completeSale`) بقي دالة يدوية عمداً — منطقه (وضع أوفلاين + طابور محلي + تراجع تفاؤلي) خاص جداً ولا يستفيد من إعادة كتابته كـ`useMutation`.
   - ⚠️ أثناء هذا التحويل اكتشفنا أننا كنا سنُفقد مهلة الـ350ms الأصلية قبل تنفيذ البحث (لأن مفتاح الاستعلام في React Query يُطلق طلباً فورياً عند تغيّره) — تم تداركها بإبقاء نفس آلية الـdebounce صراحةً قبل تمرير القيمة كمفتاح استعلام.

## Capacitor: القرار والخطة (بدل تجاهل الموضوع)

بحثنا فعلياً في التغييرات الرسمية بين Capacitor 6 و7 و8 (وثائق Capacitor نفسها) قبل اتخاذ قرار:
- **Capacitor 7** يتطلب Node 20+، Xcode 16+، Android Studio Ladybug، Java 21.
- **Capacitor 8** يتطلب Node 22+، Xcode 26+، Android Studio Otter، وحد أدنى لنشر iOS هو 15.0.
- **أهم تغيير جذري في Capacitor 8**: إعداد `android.adjustMarginsForEdgeToEdge` أُزيل نهائياً؛ التخطيط "Edge-to-Edge" صار إلزامياً، ويتطلب استخدام متغيرات CSS مثل `env(safe-area-inset-top)` بدل الاعتماد على هوامش تُضاف تلقائياً — **هذا يعني أن أي شاشة في van-app تعتمد على الهوامش التلقائية القديمة قد تحتاج تعديل CSS فعلي عند أي ترقية مستقبلية لتطبيقات أخرى نحو نفس إصدار van-app**.
- توجد أداة رسمية للترقية التلقائية بين الإصدارات: `npx @capacitor/plugin-migration-v6-to-v7@latest` ثم `v7-to-v8@latest`.

**القرار**: van-app بقي على `^8.4.2` كما كان (لم نغيّره)، والتطبيقات الثلاثة الأخرى على `^6.1.2`. لم نوحّدها جميعاً على 8 لأن ذلك يتطلب فعلياً بيئة Android Studio/Xcode أحدث واختبار كل تطبيق على جهاز حقيقي بسبب تغيير Edge-to-Edge أعلاه — وهذا خارج ما يمكن التحقق منه في هذه البيئة (بناء ويب فقط، بدون محاكي أندرويد). **الخطوة العملية التالية الموصى بها** لو قررتم التوحيد: رقّوا تطبيقاً واحداً غير van-app للتجربة (مثلاً naqaa-driver) باستخدام أداة الترقية الرسمية، اختبروه على جهاز حقيقي للتأكد من عدم وجود مشاكل Edge-to-Edge، ثم كرروا العملية على الباقي.

## ⚠️ لم يُلمس عمدًا (كما طُلب)

- **لا تعديل على قاعدة البيانات**: لا تغيير على أي RPC أو جدول أو سياسة RLS.
- **`hashPwd` (SHA256)** في أي ملف: تركناها كما هي لأن تغييرها الصحيح يتطلب تعديل RPC/الخادم، وهذا خارج نطاق "دون لمس قاعدة البيانات". النقطة موثّقة في الدراسة النقدية الأصلية.
- **باقي الشاشات** (`LoginScreen.jsx`, `TotpEnrollScreen.jsx`, `PrinterPanel.jsx`, `PrinterSettingsScreen.jsx`, `NotificationsScreen.jsx`): لم تحتوِ على منطق بيانات كافٍ يستحق فصلاً (صفر أو استدعاء واحد فقط)، فتُركت كما هي.
- **JSDoc types** أُضيفت للـhooks الرئيسية فقط (`useVanSale.js`, `useSettlement.js`) وليس كل ملف — نفس منطق التوسّع التدريجي المُتَّبع في بقية النقاط.
